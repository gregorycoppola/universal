# RLVF: Reinforcement Learning from Verified Feedback

## The Loop

1. LLM receives natural language input ("All men are mortal. Socrates is a man. Is Socrates mortal?")
2. LLM generates typed logical form:
   ```
   always [x:e]: man(theme: x) -> mortal(theme: x)
   man(theme: socrates)
   query: mortal(theme: socrates)
   ```
3. Calculus ratiocinator runs inference
4. Result: proof found (or not found, or type error, or inconsistency)
5. Reward signal: +1 for valid proof, -1 for failure, shaped rewards for partial correctness
6. LLM updates via PPO / DPO / similar

## The Reward Hierarchy

Not all verification signals are equal. A hierarchy of increasingly informative rewards:

| Level | What's checked | Reward signal |
|---|---|---|
| 0. Syntax | Does it parse? | The logical form is well-formed |
| 1. Types | Do the types check? | Roles match their declared types |
| 2. Grounding | Do entities exist in the KB? | All referenced entities are declared |
| 3. Consistency | Any contradictions? | The KB + new claims are consistent |
| 4. Inference | Does the conclusion follow? | The calculus derives the expected result |
| 5. Novelty | Is this non-trivial? | The conclusion wasn't already in the KB |

Each level gives a reward signal. An LLM that produces syntactically valid but type-incorrect logical forms gets partial credit. An LLM that produces fully verified novel inferences gets maximum reward. The hierarchy lets the LLM learn incrementally — syntax first, then types, then grounding, then inference.

## Comparison to RLHF

| Property | RLHF | RLVF |
|---|---|---|
| Reward signal | Human judgment | Formal verification |
| Foolable? | Yes | No |
| Scalable? | No (needs humans) | Yes (fully mechanical) |
| Consistent? | No (inter-rater disagreement) | Yes (deterministic) |
| Optimizes for | Appearance of correctness | Actual correctness |
| Failure mode | Confident hallucination | Honest refusal |
| Aligned with bitter lesson? | No (scales with humans) | Yes (scales with compute) |

## Comparison to Code RLVR

Code RLVR (AlphaCode, CodeRL) trains LLMs to produce code that passes tests. RLVF trains LLMs to produce reasoning that passes verification. The structure is identical:

| | Code RLVR | RLVF |
|---|---|---|
| Target language | Python, Rust, etc. | Typed logical forms |
| Verifier | Compiler + test suite | Calculus ratiocinator |
| What's verified | Functional correctness | Logical validity |
| Training data | Programming problems + tests | Natural language + expected logical forms |
| Reward shaping | Partial test passes | Reward hierarchy (syntax → types → inference) |

The key difference: code RLVR requires test cases written by humans. RLVF can generate its own test cases — because logical entailment is decidable in the forward fragment. Given a KB, the calculus can enumerate all true conclusions. Each one is a training example. The training data is *generated by the verifier itself.*

## The Self-Generating Training Data Point

This is crucial. In code RLVR, someone has to write the test cases. In RLVF:

1. Start with a KB (entities, predicates, rules)
2. The calculus enumerates all conclusions derivable from the KB
3. Each (KB, conclusion) pair is a training example: "given this KB, produce a logical form whose verification yields this conclusion"
4. The LLM learns to go from natural language descriptions of the KB to logical forms that verify

The training data grows with the KB. The KB grows with the LLM (because the LLM generates new lexicon entries and rules). The loop is self-reinforcing.
